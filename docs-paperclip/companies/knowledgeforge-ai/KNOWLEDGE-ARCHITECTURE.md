# Knowledge Architecture

> Unified reference for how KnowledgeForge AI discovers, indexes, and maintains knowledge across the Construct-AI knowledge base.

**Status:** Implemented (Phase 1)
**Last Updated:** 2026-04-06

---

## Overview

KnowledgeForge AI is responsible for reading, indexing, and maintaining the knowledge base at `docs-construct-ai/`. This document describes the complete knowledge architecture — what knowledge exists, how it's organized, how agents discover it, and how the system scales.

---

## Knowledge Layers

The knowledge base uses a 3-layer model (simplified from the original 5-layer proposal):

| Layer | Name | Purpose | Example |
|-------|------|---------|---------|
| 1 | **Reference** | Durable, read-only knowledge that defines how things work | Discipline docs, workflow guides, standards |
| 2 | **Working** | Active project knowledge that changes as work progresses | PARA projects, integration plans, implementation notes |
| 3 | **Session** | Conversation context that expires after the session ends | Agent chat history, daily memory files |

> **Note:** The original 5-layer model (A-E: Gigabrain, LCM, PARA, Daily, Session) has been simplified to 3 layers for clarity. The YAML frontmatter headers (`memory_layer`, `gigabrain_tags`, etc.) on existing documents remain valid and are used by the knowledge index.

## Knowledge Index

### What It Is

`KNOWLEDGE-INDEX.json` is a single searchable index file generated by `scripts/build-knowledge-index.py`. It maps every markdown file in `docs-construct-ai/` to its metadata, discipline, type, and tags.

### How It Works

```
scripts/build-knowledge-index.py
  ↓
Scans all .md files in docs-construct-ai/
  ↓
Extracts YAML frontmatter (memory_layer, gigabrain_tags, etc.)
  ↓
Classifies by path (discipline, workflow, agent-data, etc.)
  ↓
Outputs KNOWLEDGE-INDEX.json
```

### Index Contents

```json
{
  "version": "1.0.0",
  "summary": {
    "total_documents": 2906,
    "total_disciplines": 67,
    "total_workflows": 17,
    "total_tags": 767,
    "by_type": {
      "discipline": 639,
      "codebase": 1263,
      "workflow": 17,
      "agent-data": 109,
      ...
    }
  },
  "documents": [...],
  "disciplines": {
    "01900": {
      "name": "Procurement",
      "document_count": 3,
      "workflow_count": 0,
      "paths": [...]
    }
  },
  "workflows": {...},
  "tags_index": {...}
}
```

### How Agents Use It

1. **Doc Analyzer** runs the index script periodically to discover new/changed documents
2. **Discipline Specialists** query the index to find documents for their discipline
3. **Workflow Guardians** query the index for `workflow_docs/` paths
4. **QA Strategist** uses the index to verify documentation coverage

### Rebuilding the Index

```bash
cd docs-construct-ai
python3 scripts/build-knowledge-index.py --root . --output KNOWLEDGE-INDEX.json
```

## YAML Frontmatter Headers

Documents in `docs-construct-ai/` may include YAML frontmatter:

```yaml
---
memory_layer: durable_knowledge
para_section: pages/documentation
gigabrain_tags: documentation, procurement
openstinger_context: general-documentation
last_updated: 2026-03-30
related_docs:
  - docs/
---
```

**These headers are not technical debt — they are structured metadata** that the knowledge index uses for:
- Filtering by memory layer
- Tag-based search
- Cross-reference resolution
- Last-updated tracking

**Do not remove them.** New documents should include them when relevant.

## KnowledgeForge Agent Roles (Phase 1: 11 Core Agents)

| Agent | Role | Knowledge Sources |
|-------|------|-------------------|
| Doc Analyzer | Crawl and index all knowledge | All directories |
| QA Strategist | Review quality, generate tests | Standards, testing, codebase |
| Arch Refactorer | Propose architecture improvements | Codebase, architecture |
| App Guardian | Verify application integrity | Testing, error-tracking |
| Workflow Guardian | Maintain workflow documentation | `disciplines/*/workflow_docs/` |
| Discipline Reader | Read and summarize discipline docs | `disciplines/*/` |
| Standards Keeper | Maintain standards documentation | `standards/` |
| Memory Manager | Manage daily memory and templates | `memory-integration/`, `memory/` |
| Integration Monitor | Track integration progress | `integration/` |
| Error Tracker | Monitor and document errors | `error-tracking/` |
| Failure Learner | Learn from failures, archive old memory | All directories |

## Knowledge Sync Procedure

The `knowledge-sync` task (defined in `tasks/knowledge-sync/TASK.md`) runs periodically to:

1. Scan `docs-construct-ai/` for new or modified documents
2. Update `KNOWLEDGE-INDEX.json`
3. Report changes to the board
4. Flag documents missing frontmatter
